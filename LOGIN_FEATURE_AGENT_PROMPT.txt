You are working on a new frontend project for the Bazario backend.

Context:
- The frontend stack is Vite + React + TypeScript + Tailwind CSS + shadcn/ui.
- This is a fresh frontend, separate from the Laravel app.
- The backend already exists and should be treated as the source of truth.
- Do not use Redux unless there is a clear need. Prefer React Query for server state.
- Use clean feature-oriented architecture.
- Keep the codebase professional: strong separation of concerns, typed contracts, no ad hoc state, no duplicated auth logic.

Your task:
Implement the first production-quality auth feature: login entry flow with header integration and register navigation.

High-level behavior:
1. There should be a main layout with a header.
2. The header must show a `Login` button when the user is logged out.
3. The header must show a `Logout` button when the user is logged in.
4. Clicking `Login` should open a UI containing a login form.
5. The login form should include the required backend fields.
6. Under the form, show a text action:
   `Don't have an account yet? Register`
7. Clicking that action should navigate to the register page.
8. Login must be fully wired:
   - UI
   - form validation
   - API call
   - TypeScript types
   - auth token handling
   - logged-in state
   - logout behavior

Important architectural direction:
- Do not just build a visual form.
- Build this as a complete vertical slice.
- Keep feature code inside the auth feature unless it is truly app-wide infrastructure.
- Keep the API layer separate from page components.
- Keep backend response types explicit.
- Keep auth token handling centralized.
- Keep app-shell concerns separate from auth feature concerns.
- Use React Query for mutations and any current-user fetch if needed.
- Use react-hook-form + zod for form handling and validation.
- Use React Router for navigation.
- Use shadcn/ui components where appropriate.

Architecture boundary rules:
- `src/features/auth/**` should contain auth-specific UI, hooks, schemas, API functions, and types.
- `src/layouts/**` and `src/components/shared/**` should contain reusable app-shell concerns like the header and layout wrappers.
- `src/lib/api/**` should contain shared HTTP client infrastructure, not feature-specific request functions.
- `src/lib/auth/**` can contain token persistence and auth-session helpers that are app-wide.
- The header should consume auth state, but should not own raw login request logic.
- The login dialog/modal/sheet can live under `src/features/auth/components/**` and be rendered from the header.

Backend contract to match:

Login endpoint:
- Method: `POST`
- URL: `/api/login`
- Request body:
  - `email: string`
  - `password: string`

Login validation rules from backend:
- `email` is required and must be a valid email
- `password` is required and must be a string

Login success response envelope:
- `success: 1`
- `message: string`
- `result: { token: string; user: User; roles: string[] }`

Login failure response envelope:
- `success: 0`
- `message: string`
- `result: unknown`
- HTTP status is `401` for invalid credentials

Notes:
- The backend uses Laravel Sanctum bearer tokens.
- The frontend should store the token and send it in the `Authorization: Bearer <token>` header for protected requests.

Register endpoint:
- Method: `POST`
- URL: `/api/register`
- Request body:
  - `name: string` required
  - `age: number | null`
  - `email: string` required
  - `phone: string | null`
  - `password: string` required

Register validation rules from backend:
- `name`: required, string, max 255
- `age`: nullable, integer, min 12, max 100
- `email`: required, valid email, unique
- `phone`: nullable, string, unique
- `password`: required, string, min 6

Register success response envelope:
- `success: 1`
- `message: string`
- `result: { token: string; token_type: "Bearer"; user: User; roles: string[] }`

Register validation failure envelope:
- `success: 0`
- `message: string`
- `result.errors` contains field validation errors
- HTTP status is `422`

User model fields relevant to frontend:
- `id`
- `name`
- `email`
- `phone`
- `age`
- plus any standard timestamps if returned by backend

Role values to expect:
- `customer`
- `seller`
- `service_provider`
- `admin`
